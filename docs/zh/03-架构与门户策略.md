# 架构与门户策略

## 1. 整体建议

建议 SBBC 前期采用：

- 一个共享平台
- 一套共享后端
- 一个共享数据库
- 一套共享认证体系
- 一套共享设计系统
- 多个按角色区分的前端工作区

这是当前阶段在“隔离性”和“可维护性”之间最合适的平衡。

## 2. 为什么不建议一开始做成多个完全独立系统

如果教师端、学生端、管理端、官网从一开始就拆成完全独立的产品，通常会带来：

- 登录逻辑重复
- 用户和权限模型重复
- 部署链路更多
- 维护成本更高
- 跨模块迭代更慢
- 课程、付款、学生状态之间更容易出现同步问题

## 3. 推荐结构

### 公共网站层

- `www.sbbc.com.au`
- 品牌展示
- 课程介绍
- 文章内容
- 咨询与试听入口

### 登录后平台层

- `app.sbbc.com.au`
- 共享登录入口
- 登录后按角色跳转到对应工作区

### 角色工作区

- `app.sbbc.com.au/student/*`
- `app.sbbc.com.au/teacher/*`
- `app.sbbc.com.au/admin/*`

这能保证体验隔离，但仍保持系统整体一致。

## 4. 教师端建议

教师端建议在产品体验上独立。

也就是说：

- 独立导航
- 独立页面结构
- 独立权限边界
- 独立 API 访问控制

但在 V1 到 V3 阶段，不建议把教师端拆成完全独立的后端和数据库。

原因：

- 教师与管理员的工作目标差异很大
- 教师端需要尽量简单、专注
- 共享后端更容易和学生、课程、排课、反馈联动

## 5. 学生 Portal 建议

学生 Portal 也应作为一个独立角色工作区来设计。

建议：

- 与管理后台和教师端明显隔离
- 登录后直接进入自己的学习空间
- 共享统一认证和数据底座

学生 Portal 后续应支持：

- 已报名课程
- 剩余课时
- 我的课表
- 学习历史
- 支付信息
- 录播课程
- 直播课入口

## 6. 推荐技术栈

本节只说明建议方向，不代表所有工具都必须在第一阶段一次性接入。前期应优先选择成熟、维护成本低、方便后续扩展的方案。

### 前端

- 建议使用 Next.js 作为主框架，支持官网、后台、教师端和学生端在同一套技术体系下开发
- 前端代码建议使用 TypeScript，提高长期维护稳定性
- 页面样式建议使用 Tailwind CSS，并逐步沉淀统一的按钮、表单、表格、卡片等组件规范
- BI 看板阶段再引入合适的图表库，用于收入、招生、课程、活动等数据展示

### 后端

- 建议以 Supabase 作为基础后端，负责数据库、登录认证、文件存储和实时数据能力
- 轻量业务逻辑可以先放在 Next.js 后端能力中处理，避免早期架构过重
- 数据访问需要结合角色权限和数据库权限做隔离，确保学生、教师、助教、管理员只能访问对应数据
- 前期先不引入复杂工作流工具；后续如果自动提醒、审批、同步、重试等流程变多，再评估接入专门的自动化工作流工具

### 数据分析与搜索引擎优化（SEO）

- 基础 SEO 需要从第一阶段考虑，确保课程页、文章页和品牌页具备清晰的标题、描述和页面结构
- 建议接入 Google Search Console，用于查看 Google 搜索收录、关键词和页面表现
- 网站访问分析工具可按需要选择，例如 Google Analytics 4、Umami 或 PostHog，用于查看访问量、来源、页面表现和关键转化行为
- 自动生成网站地图（Sitemap）和搜索引擎抓取规则（Robots.txt），帮助搜索引擎正确收录页面
- 统一管理页面标题、描述、图片说明文字和结构化数据，提升搜索展示效果

### 内容层

- 课程、文章和结构化内容优先存放在应用数据库中，方便和报名、课程、学生数据关联
- 如果后续内容编辑、审核、多语言发布流程变复杂，再考虑接入更完整的内容管理系统

### 支付

- 前期建议以澳洲本地常用的在线支付能力为主，优先满足报名和课程付款流程
- 如果后续面向中文用户需要更多支付方式，再评估增加第二支付通道

### 邮件

- 邮件服务主要用于发送注册、报名、付款、课程提醒等事务邮件
- 具体邮件服务可根据成本、送达率和维护便利性选择

### 视频与直播

前期建议两条路径：

- 轻量路径：先在 Portal 中接入第三方直播链接
- 平台路径：V4 再逐步引入平台级视频与直播能力

### 托管与网络层

- Vercel 负责应用部署
- Cloudflare 负责 DNS、WAF、CDN 和域名层

## 7. 需要尽早明确的数据模型

数据模型需要和版本路线保持一致。V0/V1 不需要一次性实现所有实体，但需要先把核心关系想清楚，避免后续 CRM、教务、财务、录播等模块接入时反复推翻。

### V0/V1 必须稳定

- User：登录用户
- Role：角色与权限
- Student：学生
- Teacher：教师
- Teaching Assistant：助教或运营支持角色
- Program / Course：课程体系与具体课程
- Enrollment / Order：报名记录与订单
- Lesson Package：课时包 / 套餐
- Payment：付款记录

### V2 需要预留并逐步完善

- Parent / Guardian：家长或监护人联系信息，是否作为独立登录角色另行确认
- Lead / Prospect：意向客户记录 / 咨询线索，用于 CRM、试听预约和招生跟进
- Lesson Session：具体课次记录
- Attendance：出勤记录
- Feedback：课后反馈

### V3/V4 后续扩展

- Invoice / Receipt：发票 / 收据，配合财务基础能力
- Certificate：证书，配合自动证书生成
- Video Asset：录播视频资源，配合在线课程阶段

## 8. 安全与权限模型

核心原则：

- 角色权限控制
- 对学生和教师数据做细粒度访问限制
- 对敏感操作保留审计日志
- 内容编辑权限与教务权限分开
- 教师默认不能查看无关学生财务信息

## 9. 仓库与工程组织建议

推荐方向：

- 使用一个主仓库，必要时采用 monorepo 结构

若后续团队规模增长，可考虑：

- `apps/web`：官网和应用壳
- `packages/ui`：共享组件
- `packages/types`：共享类型
- `packages/config`：共享配置

如果前期团队较小，也可以先维持单仓库单应用，但代码结构要提前规划清楚。
