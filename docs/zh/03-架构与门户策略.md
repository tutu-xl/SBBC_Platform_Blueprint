# 架构与门户策略

## 1. 整体建议

建议 SBBC 前期采用：

- 一个共享平台
- 一套共享后端
- 一个共享数据库
- 一套共享认证体系
- 一套共享设计系统
- 多个按角色区分的前端工作区

这是当前阶段在“隔离性”和“可维护性”之间最合适的平衡。

## 2. 为什么不建议一开始做成多个完全独立系统

如果教师端、学生端、管理端、官网从一开始就拆成完全独立的产品，通常会带来：

- 登录逻辑重复
- 用户和权限模型重复
- 部署链路更多
- 维护成本更高
- 跨模块迭代更慢
- 课程、付款、学生状态之间更容易出现同步问题

## 3. 推荐结构

### 公共网站层

- `www.sbbc.com.au`
- 品牌展示
- 课程介绍
- 文章内容
- 咨询与试听入口

### 登录后平台层

- `app.sbbc.com.au`
- 共享登录入口
- 登录后按角色跳转到对应工作区

### 角色工作区

- `app.sbbc.com.au/student/*`
- `app.sbbc.com.au/teacher/*`
- `app.sbbc.com.au/admin/*`

这能保证体验隔离，但仍保持系统整体一致。

## 4. 教师端建议

教师端建议在产品体验上独立。

也就是说：

- 独立导航
- 独立页面结构
- 独立权限边界
- 独立 API 访问控制

但在 V1 到 V3 阶段，不建议把教师端拆成完全独立的后端和数据库。

原因：

- 教师与管理员的工作目标差异很大
- 教师端需要尽量简单、专注
- 共享后端更容易和学生、课程、排课、反馈联动

## 5. 学生 Portal 建议

学生 Portal 也应作为一个独立角色工作区来设计。

建议：

- 与管理后台和教师端明显隔离
- 登录后直接进入自己的学习空间
- 共享统一认证和数据底座

学生 Portal 后续应支持：

- 已报名课程
- 剩余课时
- 我的课表
- 学习历史
- 支付信息
- 录播课程
- 直播课入口

## 6. 推荐技术栈

### 前端

- Next.js
- TypeScript
- Tailwind CSS
- 基于自定义 Design System 的组件体系

### 后端

- Next.js Route Handlers / Server Actions 处理轻量业务
- Supabase 负责 Postgres、Auth、Storage、Realtime
- 若后续工作流复杂，可接 Inngest 或 Trigger.dev

### 内容层

- 课程、文章、结构化内容优先存在应用数据库中
- 若以后编辑流程很复杂，再考虑更重的 Headless CMS

### 支付

- Stripe 作为主支付通道
- 若后续面向中文用户需要更多支付方式，再补第二通道

### 邮件

- Resend 负责事务邮件

### 视频与直播

前期建议两条路径：

- 轻量路径：先在 Portal 中接入第三方直播链接
- 平台路径：V4 再逐步引入平台级视频与直播能力

### 托管与网络层

- Vercel 负责应用部署
- Cloudflare 负责 DNS、WAF、CDN 和域名层

## 7. 必须尽早稳定的数据模型

以下实体在 V0/V1 就需要认真设计，因为后面所有模块都会依赖它们：

- User
- Role
- Student
- Parent / Guardian
- Teacher
- Lead
- Program / Course
- Enrollment / Order
- Lesson Package
- Lesson Session
- Attendance
- Feedback
- Payment
- Invoice / Receipt
- Video Asset
- Certificate

## 8. 安全与权限模型

核心原则：

- 角色权限控制
- 对学生和教师数据做细粒度访问限制
- 对敏感操作保留审计日志
- 内容编辑权限与教务权限分开
- 教师默认不能查看无关学生财务信息

## 9. 仓库与工程组织建议

推荐方向：

- 使用一个主仓库，必要时采用 monorepo 结构

若后续团队规模增长，可考虑：

- `apps/web`：官网和应用壳
- `packages/ui`：共享组件
- `packages/types`：共享类型
- `packages/config`：共享配置

如果前期团队较小，也可以先维持单仓库单应用，但代码结构要提前规划清楚。
