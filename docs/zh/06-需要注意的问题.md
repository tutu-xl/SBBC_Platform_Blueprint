# 需要注意的问题

这个文档专门记录后续还需要确认、会影响架构或权限设计的问题。
如果后面还有新的注意事项，也统一补到这里，不要散落到其他文档里。

## 目前 01 文档疑问

- `Lead` 统一理解为“意向客户记录 / 咨询线索”，不是登录角色；对应的跟进能力放在 Teaching Assistant / 助教或运营支持职责下。
- 学生端需要区分成年学生和未成年学生：成年学生可以自己报名、付款并管理订单和发票；未成年学生可以发起报名 / 选择课程，但付款、确认和同意条款应由家长或监护人完成。
- 学生端和家长端的分工已经明确为：学生端偏学习过程，家长端负责未成年学生付款、条款确认、通知处理和学习监督；后续需要细化每个页面的权限边界。
- 需要把每个角色能看到的界面范围写清楚，避免后面权限边界不明确。
- 核心用户角色提到 `Helper`，但后面没有对应功能说明，需要补齐。
- `Super Admin` 的权限范围还比较抽象，建议补成更具体的功能描述。
- 现在已经提到教师端和学生端，但如果两个端的功能高度重叠，建议先说明为什么要分开设计。

## 目前 02 文档疑问

- `V4` 之前是否先不做在线课程，这一点建议再明确一下。
- 家长 / 监护人角色已经需要进入产品设计，但完整家长 Portal 的上线阶段和功能边界仍需确认。
- `V2` 里提到客服、助教、排课、教务协作，建议再补一句各自的职责边界。

## 目前 03 文档疑问

- “推荐结构”这一段写得还不够统一，建议整理成更明确的层级。
- 教师端建议写成独立入口、独立页面结构、独立权限边界、独立 API 访问控制，这样更容易理解。
- 推荐技术栈和 `04` 文档里的软件选型要统一，避免前后说法不一致。
- 之前提到的轻量路径和第三方直播链接，也建议统一表述，不要一处写一种说法。

## 目前 04 文档疑问

- 推荐采购顺序建议再校对一下，避免和前面的版本规划对不上。
- 服务采购最好和版本计划同步说明，哪些是先买，哪些是后买。
- 自动化能力建议先预留架构空间，前期先把核心业务流程跑通，后面再逐步加通知、审批、提醒和规则自动化。
- 日志是否统一存到 Supabase 需要确认，尤其是关键操作日志和审计记录。
- 数据备份需要提前纳入考虑，数据库和文件是否分别备份也要说明。

## 后续待确认问题

- 支付方式组合需要确认：是否同时考虑澳洲本地支付方式、银行卡支付，以及面向中文用户更熟悉的支付方式。
- 未成年学生的付款流程需要单独设计：未成年学生可以发起报名 / 选择课程，但付款、服务条款、隐私政策和未成年人确认事项应由家长或监护人完成。
- 家长账号需要纳入权限设计，并和学生端、付款流程、通知权限一起确定。
- 完整家长 Portal 的上线阶段需要单独商榷，目前先保证基础付款、确认、通知和学习监督流程可以成立。
- 学生端和家长端的页面权限要继续细化：学生端偏课程、课表、课时、反馈、学习资料和积分；家长端偏付款、条款确认、通知、退款、发票、账单和学习监督。
- 直播课程需要明确边界：V1 应支持一对一和小班直播授课，但前期以 Zoom、腾讯会议等第三方链接接入为主；平台自建直播能力后续再评估。
- 录播课程需要明确上线阶段：如果放到 V2 / V3 以后，是否提前预留进度追踪、测验和学习完成记录。
- 自动化能力建议先预留架构空间，让客户接受“先基础、后自动化”的节奏。
- 日志可以统一考虑放到 Supabase，但要明确哪些是普通日志，哪些是审计日志。
- 数据库和文件都建议做备份，避免后期出现数据恢复风险。
- 敏感数据访问要做权限控制，至少要说明谁能看、谁能改、谁能导出。
- `Super Admin` 需要能看到系统级配置、权限、日志和关键操作记录，但不等于日常业务管理员。

## 文档写法建议

- 版本文档里尽量写“先做什么、后做什么”，不要一下子把所有能力都堆进去。
- 客户容易接受的表达方式是：先做核心业务，再预留扩展能力。
- 如果某个功能还没确定最终方案，可以先写成“待确认项”或“后续扩展项”。
